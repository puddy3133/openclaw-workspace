# OpenClaw Cron 执行机制详解

## 🔄 任务执行时机

### 1. 正常执行
- **定时触发**: 到达设定时间，自动执行
- **条件**: OpenClaw Gateway 必须正在运行
- **时区**: 按配置的时区（Asia/Shanghai）判断

### 2. 错过执行的处理

**场景 A: 网关关闭期间错过任务**
```
情况: 任务设定 18:00 执行，但 17:50-18:10 网关关闭
结果: ❌ 任务不会补执行
原因: OpenClaw 没有"补执行"机制
```

**场景 B: 网络断开期间错过任务**
```
情况: 任务设定 18:00 执行，但网络断开
结果: ❌ 任务执行失败，不会自动重试
原因: 网络问题导致执行失败，但时间已过
```

**场景 C: 服务重启后**
```
情况: 网关重启后，检查是否有"待执行"任务
结果: ⚠️ 取决于任务类型
- systemEvent: 可能立即执行（如果配置了）
- agentTurn: 通常不会补执行
```

## ⏸️ 执行中断的处理

### 中断类型

| 中断原因 | 处理方式 | 结果 |
|---------|---------|------|
| **超时** | 强制终止 | 标记为 error，记录超时 |
| **网关重启** | 任务终止 | 标记为 error，可能部分执行 |
| **网络断开** | 任务失败 | 标记为 error，不会自动重试 |
| **手动停止** | 强制终止 | 标记为 error |

### 错误处理

**查看错误状态**:
```bash
openclaw cron list
# 查看 lastStatus、consecutiveErrors、lastError
```

**错误示例**:
```json
{
  "lastStatus": "error",
  "lastDurationMs": 600054,
  "consecutiveErrors": 2,
  "lastError": "Error: cron: job execution timed out"
}
```

## 🛡️ 可靠性保障建议

### 1. 幂等性设计

**好的设计**:
- 任务可以安全地多次执行
- 检查状态后再执行操作
- 记录执行日志，避免重复

**示例**:
```python
# 每日日志创建任务
if not os.path.exists(f"memory/{today}.md"):
    create_log_template()
else:
    print("今日日志已存在，跳过")
```

### 2. 关键任务双保险

**方案 A: Heartbeat 补偿**
- 在 HEARTBEAT.md 中添加检查
- 每次心跳时检查是否有遗漏任务

**方案 B: 启动时检查**
- 在 AGENTS.md 启动流程中添加
- 检查昨日/上周任务是否完成

**方案 C: 手动补救**
- 发现遗漏时，手动触发执行
```bash
openclaw cron run <job-id>
```

### 3. 任务设计原则

| 原则 | 说明 |
|------|------|
| **短任务优先** | 避免长时间运行的任务 |
| **可中断** | 任务可以被安全中断 |
| **可重试** | 失败后手动重试无副作用 |
| **有日志** | 记录执行状态便于排查 |

## 🔧 改进建议

### 1. 添加启动检查

在 `AGENTS.md` 或 `HEARTBEAT.md` 中添加：
```markdown
## 启动时检查
- [ ] 检查昨日日志是否创建
- [ ] 检查上周回顾是否生成
- [ ] 如有遗漏，提醒用户手动触发
```

### 2. 设置任务超时

合理设置 `timeoutSeconds`:
- 简单任务: 60 秒
- 复杂任务: 180 秒
- 避免无限等待

### 3. 监控错误率

关注 `consecutiveErrors`:
- 连续错误 > 3 次，需要排查
- 手动修复后重置错误计数

## ❓ 常见问题

**Q: 任务错过了怎么办？**
A: 手动执行 `openclaw cron run <job-id>`

**Q: 如何查看任务执行历史？**
A: `openclaw cron runs <job-id>`

**Q: 可以设置任务重试吗？**
A: OpenClaw 没有自动重试，需要手动或 Heartbeat 补偿

**Q: 任务执行时可以查看日志吗？**
A: 可以，`openclaw logs --follow`
