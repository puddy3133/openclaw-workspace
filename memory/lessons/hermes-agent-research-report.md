## 一、Hermes 是什么？

**Hermes Agent** 是由 **Nous Research** 开发的开源 AI Agent 框架，核心卖点是**「自进化学习能力」**——它能从经验中创建技能、在使用中改进技能、跨会话构建用户模型。

**关键数据**：
- GitHub Stars: **64.1k**
- Forks: **8.6k**
- Commits: **3956+**
- 开源协议: **MIT**

**与 OpenClaw 的关系**：Hermes 实际上是 OpenClaw 的「进化版」。从 GitHub 提交记录可以看到大量 `rebrand OpenClaw references to Hermes` 的改动，并且提供了 `hermes claw migrate` 命令专门从 OpenClaw 迁移。

## 二、核心特性分析

### 2.1 学习闭环（Learning Loop）

这是 Hermes 与其他 Agent 框架（包括 OpenClaw）最大的区别：

```
任务执行 → 结果评估 → 技能提取 → 技能精炼 → 技能检索
    ↑                                              ↓
    └──────────────── 下次遇到类似问题 ──────────────┘
```

**三个核心属性**：

| 属性 | 说明 |
|------|------|
| **技能创建** | 成功完成任务后，自动提取推理模式为可复用技能 |
| **技能改进** | 新证据到达时更新技能；用户偏好变化时自适应 |
| **用户建模** | 跨会话构建用户画像（偏好、决策历史、任务模式） |

### 2.2 用户建模（User Modeling）

Hermes 跨会话追踪四类信息：

1. **任务偏好**：输出格式、结构、风格偏好
2. **决策历史**：用户在类似情况下的选择
3. **常见任务模式**：任务类型、上下文、频率
4. **反馈信号**：显式修正和隐式接受模式

**价值**：用户不需要每次都重复偏好，Agent 会「记住」你的习惯。

### 2.3 多平台支持

**入口**：
- CLI（终端 UI）
- Telegram
- Discord
- Slack
- WhatsApp
- Signal
- Email

**终端后端**：
- Local（本地）
- Docker
- SSH
- Daytona（无服务器）
- Singularity
- Modal（无服务器）

### 2.4 其他特性

- **Cron 定时任务**：自然语言定义定时任务
- **子 Agent 并行**：派发隔离子 Agent 并行执行
- **MCP 集成**：支持连接任意 MCP 服务器
- **技能系统**：兼容 [agentskills.io](https://agentskills.io) 开放标准
- **模型自由切换**：支持 Nous Portal、OpenRouter（200+ 模型）、z.ai/GLM、Kimi、MiniMax、OpenAI 等

## 三、与 OpenClaw 对比

| 特性 | Hermes | OpenClaw |
|------|--------|----------|
| 技能创建 | ✅ 从经验自动创建 | ❌ 无原生支持 |
| 技能改进 | ✅ 持续精炼 | ❌ 无 |
| 跨会话用户建模 | ✅ 完整支持 | ⚠️ 有限 |
| 响应式工具使用 | ✅ | ✅ |
| 多 Agent 支持 | ✅ | ✅ |
| 开源 | ✅ | ✅ |
| **架构复杂度** | 中等 | 低 |
| **部署速度** | 较慢 | 快 |

**设计哲学差异**：

- **OpenClaw**：追求广泛的工具覆盖和快速部署，无跨会话学习
- **Hermes**：牺牲一定的部署简便性，换取长期改进能力

## 四、值得学习的点

### 4.1 技能系统设计

**Hermes 的技能概念**：
- 技能是「结构化推理模板」
- 「当上下文看起来像这样时，这个方法有效」
- 技能可以是简单的（输出格式）或复杂的（多步推理策略）

**我们可以借鉴**：
- 将任务执行成功后的「推理链」抽象为模板
- 按任务类型分类存储技能
- 下次遇到类似任务时优先检索相关技能

### 4.2 用户建模思路

**Hermes 的四维度模型**：
```
用户模型 = {
  任务偏好: { 输出格式, 结构, 风格 },
  决策历史: { 相似情况下的选择 },
  任务模式: { 类型, 上下文, 频率 },
  反馈信号: { 显式修正, 隐式接受 }
}
```

**我们可以借鉴**：
- 在 `USER.md` 基础上扩展用户偏好追踪
- 记录「为什么用户修改了我的输出」
- 自动归纳用户偏好变化趋势

### 4.3 学习闭环机制

**Hermes 的学习触发条件**：
- 任务成功 + 方法非平凡 → 提取技能
- 新方法持续优于旧技能 → 更新技能
- 用户偏好变化 → 自适应

**我们可以借鉴**：
- 任务完成后自检：「这个方法值得记录吗？」
- 对比历史：「上次怎么做的？这次更好吗？」
- 用户反馈：「用户修改了什么？为什么？」

### 4.4 架构设计

**Hermes 的模块化**：
```
hermes-agent/
├── agent/           # 核心推理
├── gateway/         # 多平台消息网关
├── skills/          # 技能系统
├── tools/           # 工具集
├── cron/            # 定时任务
├── plugins/         # 插件系统
├── acp_adapter/     # ACP 协议适配
└── environments/    # RL 环境
```

**我们可以借鉴**：
- Gateway 独立进程设计（支持多平台同时在线）
- 插件化工具系统
- ACP 协议支持（与 Claude Code 等 ACP 客户端互通）

### 4.5 迁移友好度

Hermes 提供了完整的 OpenClaw 迁移方案：
- 自动检测 `~/.openclaw`
- 导入 SOUL.md、MEMORY.md、技能、配置
- 支持 dry-run 预览

**我们可以借鉴**：
- 如果未来要重构架构，保留迁移路径
- 版本升级时提供自动迁移脚本

## 五、适用场景分析

### 适合 Hermes 的场景

1. **重复性结构化任务**：类似问题反复出现
2. **固定用户/团队长期使用**：用户建模才有价值
3. **可衡量改进效果**：能观察到 Agent 越用越好

### 不适合 Hermes 的场景

1. **一次性广泛任务**：无重复模式，技能库用不上
2. **基础设施能力有限**：需要自托管技能存储、学习循环
3. **用户高度分散**：成千上万用户各用一次，用户建模无效

## 六、对我们当前架构的启发

### 6.1 可立即借鉴

| 概念 | 当前状态 | 改进方向 |
|------|----------|----------|
| 技能系统 | 有 Skill 系统，但无「从经验创建」 | 任务成功后自动生成 Skill 草稿 |
| 用户模型 | USER.md + 对话中学习 | 扩展偏好追踪维度 |
| 记忆系统 | MEMORY.md + 每日日志 | 增加跨会话检索能力 |

### 6.2 需要评估

- **Gateway 独立进程**：目前 OpenClaw 已有 Gateway，是否需要增强？
- **学习闭环**：需要多大的架构改动？
- **多 Agent 协作**：目前有子 Agent 系统，是否需要增强？

### 6.3 暂不考虑

- **RL 训练环境**：目前不需要
- **无服务器后端**：本地部署足够
- **完整迁移到 Hermes**：OpenClaw 生态已成熟，迁移成本高

---

## 七、结论

Hermes 是一个**理念先进、工程成熟**的 Agent 框架，其「学习闭环」和「用户建模」机制值得深入学习。

**核心启示**：
> Agent 不应该只是「执行任务」，而应该是「越用越懂你」、「越用越熟练」。

**建议行动**：
1. 研究 Hermes 技能系统的具体实现
2. 在现有记忆系统上增加「技能提取」机制
3. 扩展用户建模维度，从被动记录转向主动学习
